# GPU プローブ結果 2026-07-21

段階3（Adreno + Vulkan）の最初の関門「compute 対応デバイス／キューが露出しているか」の実行時プローブ結果。
**合格。** ネイティブ Adreno ドライバが compute 可能なキューを露出している。

## 実施方法

```powershell
vulkaninfo --summary          # デバイス一覧
vulkaninfo                    # 全リミット（キューファミリ・subgroup・メモリ）
```

Vulkan SDK 1.4.350.0、ローダ `vulkan-1.dll`（`C:\Windows\System32`）。Instance version 1.4.350。

## 検出されたデバイス

| GPU | 名称 | 種別 | 位置づけ |
|---|---|---|---|
| GPU0 | Qualcomm Adreno X1-85 | INTEGRATED_GPU、ネイティブ Qualcomm 専用ドライバ | **本命** |
| GPU1 | 同 Adreno（Microsoft Direct3D12 経由） | D3D12 マッピングレイヤー | 変換オーバーヘッド大、非採用 |

CUDA 非依存・ノート単体という方針では **GPU0（ネイティブ Adreno）を対象**とする。

## 判定と、その先のリスク（重要）

**プローブ合格は「動かせる」であって「速い」ではない。** 拾ったリミットが、既存の Rémy サンプラーを
GPU にそのまま移植した場合のリスク（CPU に負ける可能性）を、むしろ具体的に裏づけた。

- **subgroupSize = 64**: 64スレッドがロックステップ。Rémy 木の再帰（`ViolationSampler.countEars` /
  `leafCount`）は木の形でスレッドごとに経路が割れ、この64幅の中で分岐発散して実効並列度が落ちる。
- **共有メモリ 32 KB / ワークグループ**: n=60 で1スレッドの木の状態は約3KB（int配列6本＋bool配列）。
  高速な共有メモリに載せようとすると `32KB ÷ 3KB ≈ 10スレッド`しか入らず、64幅の subgroup すら埋められない。
  プライベートメモリに逃がすと 15.78GB のグローバルメモリへスピルし、メモリ遅延律速になる。

結論として、CPU 実測（n=60 で 288k 標本/秒、12コア）に対し **Rémy 素朴移植は勝てない見込み**。
GPU で勝つには、ポインタ追跡と大状態を持たない **unranking（id→三角形分割の直接構成）への設計変更**が要る。
ただし unranking は Cₙ が 64bit を溢れるため n≲30 に制約され、規則自体が新規設計（要検討）。

## 最小 compute カーネル：実装・実証済み（2026-07-21）

プローブの次の足場として、GPU 上で `result = mix64(seed ^ mix64(index))` を全スレッド分計算させ、
CPU 側の `SplitMix64.mix64` と1要素ずつ突き合わせる最小 compute カーネルを実装した。**全一致で成功。**

```
Generated SPIR-V (2060 Bytes)
Selected device: Qualcomm(R) Adreno(TM) X1-85 GPU
PROBE OK: All results (1000000 elements) are matched between GPU and CPU.
```

- **Java → SPIR-V → Vulkan → Adreno のツールチェーン全体が機能**することを、正しさ検証つきで実証。
  100万スレッドの GPU 計算が CPU 基準とビット単位で全一致。
- SPIR-V は shaderc（LWJGL バンドル）で**実行時**にコンパイルするため、Vulkan SDK が無い環境でも
  Maven の依存だけで再現可能。GLSL 側は `shaderInt64`（実機対応済み）で 64bit 演算を直接記述。
- 実行には `--enable-native-access=ALL-UNNAMED` を付けて LWJGL の警告を抑止（動作自体には不要）。

実装（`dev.shoheiyamagiwa.enumerator.gpu` パッケージ、責務分割）:

| クラス | 責務 |
|---|---|
| `ShaderCompiler` | GLSL → SPIR-V（shaderc、実行時） |
| `VulkanContext` | Vulkan 環境（インスタンス・物理デバイス選択・compute キュー・コマンドプール）、`shaderInt64` 有効化 |
| `StorageBuffer` | GPU バッファの確保・破棄・読み戻し（ホスト可視コヒーレント） |
| `Mix64ComputePipeline` | mix64 用パイプライン一式（シェーダモジュール・ディスクリプタ・プッシュ定数） |
| `Mix64GpuRunner` | ディスパッチの記録・投入・完了待ち・読み戻し |
| `Mix64GpuProbe` | 配線と CPU 一致検証（エントリポイント） |
| `resources/shaders/mix64.comp` | GLSL カーネル本体 |

## 次の一手

最小カーネルが通り、「動かせる」ことは実証できた。ただし本題は「Rémy サンプラーを GPU で CPU より速く回せるか」で、
上記「判定と、その先のリスク」のとおり素朴移植は勝てない見込み。次は **unranking（id→三角形分割の直接構成）** の
設計検討に進む。Cₙ が 64bit を溢れる n≲30 制約の中で、ポインタ追跡と大状態を持たない GPU 向き構成が成り立つかを詰める。
