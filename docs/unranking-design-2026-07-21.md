# unranking サンプラー設計メモ 2026-07-21

GPU（Adreno + Vulkan）で設計空間サンプリングを CPU より速く回すための、Rémy に代わる
**unranking（id→三角形分割の直接構成）** 方式の設計検討。実装前の判断材料として残す。

## なぜ Rémy ではなく unranking か

GPU プローブ（`gpu-probe-2026-07-21.md`）で確認したハード制約が、Rémy 素朴移植のリスクを裏づけた:
subgroupSize=64（分岐発散のペナルティ単位）、共有メモリ 32KB/ワークグループ。Rémy は
1スレッドあたり ~3KB の可変状態（int 配列6本＋bool）＋ポインタ追跡＋乱数による木の継ぎ替え＋再帰、で
GPU が最も苦手とする形。CPU 実測（n=60 で 288k 標本/秒、12コア）に素朴移植では勝てない見込み。

unranking は次の点で GPU 向き:

- **ポインタ追跡がない**: インデックス t からカタラン数テーブルの参照だけで三角形分割を構成する。
- **状態が小さい**: 明示スタック `(lo, hi, subIndex)` の O(n) 段のみ（後述、n=35 で約 560 バイト）。
- **制御フローが揃う**: 全スレッドが同じ小さなカタラン表を走査する。発散が限定的で規則的。
- **カウンタベースと相性が良い**: t は `(seed, sampleIndex)` から決定的に引ける。既存の列挙器の
  `id ∈ [0, N)` 思想とも一致する。

## アルゴリズム概要（耳数を降下中に計算できる）

三角形分割 ↔ カタラン構造。頂点 [lo..hi] の凸多角形（底辺 (lo,hi)）の三角形分割を、インデックス
t ∈ [0, C_size) から次のように構成する（size = 三角形数 = (hi-lo)-1）:

1. 明示スタックに根問題 `(lo=0, hi=n+1, subIndex=t)` を積む。
2. スタックから `(lo, hi, idx)` を取り出す。三角形数 = (hi-lo)-1。0 なら辺なのでスキップ。
3. 頂点 m ∈ [lo+1 .. hi-1] を選ぶ。左 [lo..m] の三角形数 = m-lo-1、右 [m..hi] = hi-m-1。
   apex を m にした分割の個数 = C_left × C_right。m を走査して idx から順に引き、
   idx がブロックに収まった m を採用。leftIndex = idx / C_right、rightIndex = idx % C_right。
4. 三角形 (lo, m, hi) を確定。**耳判定はこの場で可能**: 辺 (lo,m),(m,hi),(lo,hi) のうち
   2つ以上が境界辺なら耳（`TriangulationUtils.isBoundary(a, c, n+2)` と同じ判定）。耳カウンタを進める。
5. `(lo, m, leftIndex)` と `(m, hi, rightIndex)` をスタックに積む。

現行のプレースホルダ評価器は `violations = setDirectionBits + (ears - 2)` なので、**カーネルが必要とするのは
耳数だけ**。木を丸ごと materialize せず、降下中に耳を数えれば済む。

- **カタラン表**: C_0..C_35 の 36 個 × 8 バイト = 288 バイト。読み取り専用・全スレッド共有で
  uniform buffer / 定数メモリに載る。理想的。
- **スタック**: 最悪深さ ~n（caterpillar 型分割）。n=35 で 35 段 ×（idx 8B + lo 4B + hi 4B）≈ 560 バイト。
  Rémy の ~3KB より大幅に小さく、ポインタ追跡も木の継ぎ替えもない。

## 64bit の天井と CPU/GPU の役割分担

C_n と設計総数の 64bit 符号付き（2^63-1）到達点（実測、`catalan.py`）:

| | 64bit に収まる最大 n |
|---|---|
| C_n（三角形分割インデックス単体） | **n ≤ 35**（C_35 ≈ 3.1×10^18） |
| 2^(n-1)·C_n（三角形分割×向きの合成インデックス） | n ≤ 23 |

**含意: t（三角形分割）と o（向き）を合成した単一インデックスにせず、分離して扱うこと。**
分離すれば天井は C_n 律速の **n ≤ 35**。合成すると n ≤ 23 に落ちる。向き o は n-1 ビット
（n≤35 なら 34 ビット）なので 64bit 1語に収まり、合成の必要はそもそもない。

したがって役割分担は明快:

- **小〜中 n（n ≤ 35）**: GPU で unranking。
- **大 n（n > 35）**: CPU で Rémy（64bit を溢れる領域はサンプリング前提のまま）。

これは弱点ではなく、**手法の適用範囲を実測で示す**という発表の筋になる。

## 未解決の設計論点（実装前に詰める）

1. **評価規則との結合**: 上記はプレースホルダ（耳数のみ）前提。本番デメテル評価器（三角形→デルタ規則が
   確定した後）は、各三角形の受け手参照の型を見る必要があり、カーネルが必要とする情報が変わる。
   ただし**プレースホルダ評価器に対する unranking は今すぐ完全に仕様化でき**、GPU-vs-CPU の
   サンプリング＋評価スループットを測れる（強スケーリングベンチの GPU 版に相当）。本番規則が決まったら
   カーネル内の三角形評価だけ差し替わる。→ 三角形→デルタ規則の確定を待たずに GPU 実測に入れる。

2. **一様な t ∈ [0, C_n) の生成**: カウンタベース RNG で 64bit を引き、[0, C_n) へ一様に縮小する必要がある。
   Lemire の乗算上位法は 64×64→128bit を要し、GPU では `shaderInt64` でも 128bit 積の合成が要る。
   棄却法なら 128bit 不要だがループ（発散）が入る。低棄却率だが要検討。solvable だが非自明。

3. **検証戦略の変更（重要）**: Rémy と unranking は**どちらも一様だが、RNG 描画を三角形分割へ写す規則が
   異なる**。同じ `(seed, sampleIndex)` でも生成される三角形分割が違うため、`FastPathEquivalenceTest` の
   「標本ごとの一致」で Rémy 側と橋渡しできない。unranking には次の2段の検証が要る:
   - (a) **unranking-CPU == unranking-GPU**（標本ごとにビット一致）。GPU 正しさの検証。mix64 プローブと同型。
   - (b) **unranking の統計 == 厳密解の統計**（satFrac の収束、小 n では t∈[0,C_n)×o∈[0,2^(n-1)) の
     全列挙が `ExactDesignEvaluator` の厳密カウントを再現）。一様性を厳密列挙器に錨付けする。

   つまり unranking は**新しい遅い基準（unranking-CPU）** を持ち、その一様性は Rémy ではなく厳密列挙器で裏取りする。

## 推奨する進め方

1. **CPU 側に unranking サンプラーを先に実装**する（`UnrankingTriangulationSampler` 等）。
   - 一様性を `ExactDesignEvaluator` で裏取り（論点3-b）。Vulkan に触れる前にここを固める。
   - これ自体が GPU 版の「遅い基準」になる（プロジェクトの規律: 速い実装を遅い基準で検証）。
   - 既存の `ViolationSampler` ↔ `NaiveViolationSampler` と同じ関係を、unranking 世界で作る。
2. **その CPU 実装を GPU カーネルへ移植**し、(a) の標本ごとビット一致を mix64 プローブと同型に検証。
3. GPU-vs-CPU のスループットを測り、n ≤ 35 の範囲で Rémy-CPU（288k 標本/秒）を上回れるか実測で判定する。

論点1のとおり、この筋は三角形→デルタ規則の確定を待たずに（プレースホルダ評価器のまま）進められる。
規則が決まれば、CPU/GPU 両方のカーネル内の三角形評価だけを同じ定義へ差し替える。
