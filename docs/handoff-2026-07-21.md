# 引き継ぎ・総まとめ 2026-07-21

プレゼンテーション作成と次セッション引き継ぎのための総括。プロジェクトの目的・これまでの変遷・
確定した結果・未確定の宿題を1枚にまとめる。詳細は各 `docs/*.md` と `CLAUDE.md` を参照。

---

## 0. 一言で

**デルタ代数に基づく「振る舞い等価な設計」の列挙・サンプリングを、並列計算（マルチコア / SIMD / GPU）で
高速化・大規模化し、その効果を JMH で厳密に実測したプロジェクト。** 大学院の並列計算（AVX/CUDA）講義の
最終発表が題材。実行環境はノート（Windows ARM64、Snapdragon X Elite、JDK 25 arm ネイティブ）で完結。

## 1. 背景と理論（プレゼンの導入に使う）

- デルタ代数：参照構造の合成でオブジェクト指向設計を表す。合成デルタは幾何的に多角形をなし、
  **境界（多角形の最外周の参照関係）が同一なら、内部の三角形分割に依存せず振る舞いが等価**であることが保証される。
- 逆に、同じ境界でも**三角形分割の違いが異なる設計を表す**。→ 境界多角形の三角形分割を数えることが、
  振る舞い等価な設計選択肢の列挙になる。
- 評価軸は**デメテルの法則**：「メソッド内では、フィールドまたは引数で参照できるクラスのメソッドのみ呼べる」。
- **設計空間の爆発**：原始開デルタ n 個に対し設計数は `2^(n-1)·C_n`（C_n はカタラン数）。
  n=12 で約4.3億、n=15 で約1600億。厳密全列挙はメモリ律速で n≤12〜13、64bit 整数も大 n で溢れる。
- → 大 n はサンプリング前提。Rémy による一様ランダム三角形分割＋カウンタベース RNG（`(seed,i)` から決定的）で、
  **並列結果が直列と一致することを検証可能に保つ**のが設計の肝。

## 2. アーキテクチャ（4層）と実装状況

| 層 | 内容 | 状況 |
|---|---|---|
| 抽出 (A) | 設計→原始デルタ分解・境界多角形 | 未実装。当面は境界多角形を抽象表現として直接入力 |
| 列挙 (B) | (n+2)角形の三角形分割 C_n 通り × 向き割当 2^(n-1) 通りを id∈[0,N) で生成 | **実装・検証済み** |
| 評価 (C) | 1設計→違反数の純粋関数 | 仕様経路は実装済み／高速経路はプレースホルダ（下記） |
| 実装化 (D) | 選ばれた設計のコード生成 | 未実装（発表では少数のみ想定） |

**評価器の二重実装（重要）**：
- 仕様経路 `core.DeltaDesignEvaluator.evaluateLawOfDemeter`：**実装済み**。受け手参照が LOCAL/TEMPORARY なら違反。
- 高速経路 `core.DesignEvaluator.violations`：**プレースホルダのまま**。構造のみの `setDirectionBits + (ears - 2)`。
- **両者はまだ同じ定義になっていない。** 高速経路への本番規則の反映は宿題（§6）。

## 3. 変遷（コミット順の物語）

到達順にシングル → マルチコア → NEON → 本番評価器 → 測定安定化 → GPU、と進んだ。

1. **評価器と代数の土台**：`DeltaDesignEvaluator`（デメテル判定）を実装。`Class` / `Reference` を不変化し
   `hashCode` を追加（合成時に `HashSet`/`LinkedHashSet` で重複除去が壊れる欠陥を修正）。
2. **測定側の不変条件を固定**：`FastPathEquivalenceTest` を追加。高速経路と遅い基準の一致を自動検証
   （§5）。同時に `Sampler` の隠れ状態 `gk` を削除し引数化（未設定だと違反数が黙って狂う罠を除去）。
3. **命名・責務の整理**：`Acc`→`Accumulator`、略記変数の展開、`DesignSampler` を
   `RemyTriangulationSampler` / `NaiveViolationSampler` / `ExactDesignEvaluator` に分割。
4. **JMH 導入**：手書き `System.nanoTime()` 計測を JMH に置換。注釈プロセッサを `pom.xml` で明示有効化
   （JDK 23+ は既定で無効）。3本のベンチと `JmhRunner` を追加。
5. **測定の安定化**：初回測定（07-20）は誤差最大 ±43% で不安定。`-f 5`・ヒープ固定・`-prof gc`・
   測定時間延長で取り直し（07-21）。**ゼロアロケーションの結論が逆転**（§4）。
6. **GPU 技術検証**：Vulkan プローブ → 最小 compute カーネル（mix64）を実装し、GPU 結果が CPU と全一致。
   LWJGL 3.4.2（`natives-windows-arm64`）＋ shaderc。`VulkanResults` に成功判定を集約。

## 4. 確定した測定結果（`docs/benchmark-2026-07-21.md` が正）

測定環境：Snapdragon X Elite (12コア)、JDK 25.0.1 arm、JMH 1.37、`@Fork(5)`・warmup 5×3s・measurement 10×3s。

### 結果1: ゼロアロケーション化（最も強い結果）

| n | 遅い基準 (ops/s) | 高速経路 (ops/s) | 高速化 | 遅い基準 alloc | 高速経路 alloc |
|---:|---:|---:|---:|---:|---:|
| 12 | 746,623 ± 17,383 | 1,538,632 ± 1,550 | **2.06x** | 3,709 B/op | ~0 |
| 30 | 231,448 ± 682 | 589,754 ± 563 | **2.55x** | 8,992 B/op | ~0 |
| 60 | 97,780 ± 105 | 288,946 ± 150 | **2.96x** | 17,816 B/op | ~0 |

- 高速化は n とともに拡大（2.06→2.96倍）。`gc.count=0` で**割り当てゼロを直接実証**。
- **教訓**：初回（不安定）測定では n=12 で「遅い基準の方が速い」と誤結論していた。安定化で逆転。
  相互検証（並列 t=1 の 287k 標本/秒 ≈ 独立ベンチ 289k ops/s、0.4%差）で新数値の整合を確認。

### 結果2: SIMD（因果を確定）

| 参照数 | 通常 scalar | 通常 vector | S/V | 自動ベクトル化OFF S/V |
|---:|---:|---:|---:|---:|
| 1024 | 11.187 | 10.360 | 1.08 | 1.31 |
| 4096 | 27.076 | 26.719 | 1.01 | 1.67 |
| 16384 | 91.784 | 89.394 | 1.03 | 1.81 |
| 65536 | 373.082 | 369.290 | 1.01 | **1.82** |

- 対照実験（`-XX:-UseSuperWord`）では参照集合が大きいほど NEON 理論上限2倍に単調接近、**1.82倍で頭打ち**。
- scalar を ON/OFF 比較すると 1.2〜1.3倍差 → **C2 がスカラーループを自動ベクトル化している動かぬ証拠**。
- **主張**：「Vector API で速くなる」ではなく、**「NEON で 1.82倍出せるが、既定構成では C2 の自動ベクトル化が
  同じ仕事をするため差が出ない。Vector API の価値は速度でなく、自動ベクトル化の適用可否に依存せず
  SIMD を保証できること」**。

### 結果3: 強スケーリング（数値は堅いが解釈に注意）

| threads | ms/op | speedup | efficiency |
|---:|---:|---:|---:|
| 1 | 695.03 | 1.00 | — |
| 2 | 348.21 | 2.00 | 0.998 |
| 4 | 175.00 | 3.97 | 0.993 |
| 8 | 100.38 | 6.92 | 0.866 |
| 12 | 82.22 | 8.45 | 0.704 |

- 2〜4スレッドで効率 ~0.99（embarrassingly parallel を裏づけ）。4スレッドを境に崖状に低下。
- **崖は同期コストではなく全コア稼働時の周波数スケーリングが支配的と見られる**（崖の位置が全コア稼働閾値と整合）。
- **効率は単一値で語れない**（基準 t=1 のブースト状態に依存）。CLAUDE.md の旧「~0.97」は撤回済み。
- 残課題：周波数固定 or クロック実測で確定（§6）。

## 5. 検証の規律（プレゼンの「正しさ」パート）

**「速い実装を、独立な遅い基準で検証してからコミットする」**を `FastPathEquivalenceTest`（9テスト）で自動化：

- `ViolationSampler`（ゼロアロケ高速経路）== `NaiveViolationSampler`（遅い基準）：n=2..12 × 各200標本で標本ごと一致。
- 並列 == 直列：best / satisfying / ヒストグラム全体を 2,3,4,8 スレッドで照合。端数チャンクも検証。
- `DesignEnumerator` == `ExactDesignEvaluator`：独立な2実装の厳密解が n=2..8 で一致。
- RNG 描画順は `ViolationSampler.eval` と `NaiveViolationSampler.sampleViolations` で一致させる（順序を変えると
  一様性を保ったまま標本が食い違う）。

## 6. GPU（Adreno + Vulkan）の到達点

`docs/gpu-probe-2026-07-21.md` に詳細。

- **プローブ合格**：ネイティブ Adreno X1-85（Qualcomm 専用ドライバ、Vulkan 1.3、compute キューあり、
  subgroupSize=64、共有メモリ 32KB/WG、shaderInt64=true、15.78GiB ユニファイドメモリ）。
- **最小 compute カーネル実証済み**：GPU で `mix64(seed ^ mix64(index))` を100万スレッド計算し、
  CPU の `SplitMix64.mix64` と**全要素ビット一致**。Java → shaderc → SPIR-V → Vulkan → Adreno の
  ツールチェーン全体が、正しさ検証つきで機能。SDK 不要（実行時 shaderc）で再現可能。
  実行時は `--enable-native-access=ALL-UNNAMED` を付す。
- **`gpu` パッケージ**（責務分割）：`ShaderCompiler` / `VulkanContext` / `StorageBuffer` /
  `Mix64ComputePipeline` / `Mix64GpuRunner` / `Mix64GpuProbe` / `VulkanResults` ＋ `resources/shaders/mix64.comp`。

### GPU の設計上のリスクと unranking（`docs/unranking-design-2026-07-21.md`）

- **Rémy サンプラーの素朴な GPU 移植は、CPU（288k 標本/秒）に勝てない見込み**：スレッドあたり ~3KB の状態、
  ポインタ追跡、木の継ぎ替え、再帰。subgroupSize=64 での分岐発散と 32KB 共有メモリ制約が裏づける。
- **打開策 = unranking**：id→三角形分割の直接構成。カタラン表（288B）＋明示スタック（n=35 で ~560B）で、
  ポインタ追跡なし・状態小・制御フロー規則的。GPU 向き。既存の id∈[0,N) 思想とも一致。
- **64bit 天井**：C_n は n≤35、合成インデックス 2^(n-1)·C_n は n≤23（実測 `catalan.py`）。三角形分割インデックス t と
  向きビット o を**分離すれば天井は n≤35**。→「小〜中 n は GPU/unranking、大 n は CPU/Rémy」。

## 7. 未確定の宿題（次にやること）

**要石は「三角形→原始デルタ規則」の確定。これが接続アダプタ・高速経路の本番化・GPU 比較の意味づけを
同時に解錠する。**

1. **三角形→デルタ規則**（研究判断、新田先生・論文の定義が要る）：
   - 各三角形の3辺を役割 (def / 受け手 r1 / r2) に割り当てる規則を定める。
   - **耳（境界辺2本＋対角線1本）は def が一意に決まり局所的**。対角線を複数持つ三角形は def/use が非一意で、
     向きビット＋導出順序で解く。**それが局所（per-triangle）か大域（ルート→リーフ順序）かが未確定**。
   - 純粋関数の型は `Polygon → Delta` では不足。`(Polygon, Triangulation, 向きビット) → List<PrimitiveDelta>`
     （評価だけなら `combine` 不要）。局所規則なら並列化が素直、大域なら律速が列挙側に移る。
2. **接続アダプタ**：抽象設計（三角形分割＋向き＋境界）→ `PrimitiveDelta` 列 → 評価。規則確定が前提。
3. **高速経路への評価規則の反映**：`DesignEvaluator.violations` を仕様経路と同じ定義に揃える。
   なお確定した判定規則は原始デルタ1個あたり参照型1つ見るだけで極めて軽く、「1設計あたりが重い純粋関数」
   という並列化の前提が崩れる。律速が列挙側に移る点をどう扱うか要検討。
4. **強スケーリング効率の周波数要因の確定**：クロック測定 or 固定。
5. **GPU 比較（検討中の案）**：n≤12 で CPU-single / CPU-multi / GPU の厳密全列挙を比較する案。
   **注意点**：現行プレースホルダ評価器だと、向き 2^(n-1) にわたる集計が閉じた式（シフト二項分布）になり、
   CPU 最適が閉じた式に逃げてハード比較として崩れる。逃げ道は (a) 本番評価器を待つ（向きごとに違反が変わり
   全列挙が本物の仕事になる）、(b) 3プラットフォームで同一アルゴリズムを総当たり強制し「N 個の独立設計の
   スループット比較」と枠組みを明示する、のいずれか。「最適な手法で比較」は前者が要る。

## 8. プレゼンの推奨ストーリーライン

1. **問題**：デルタ代数の設計空間爆発（2^(n-1)·C_n、指数増）。厳密列挙の限界。
2. **方針**：サンプリング化（Rémy＋カウンタベース RNG）＋並列化。検証可能性（並列==直列）を担保する設計。
3. **正しさの規律**：速い実装を遅い基準で自動検証（`FastPathEquivalenceTest`）。
4. **結果①ゼロアロケーション**：2.06〜2.96倍、GC ゼロを実証。**測定安定化で結論が逆転した教訓**も含めると誠実。
5. **結果②SIMD の因果**：NEON 1.82倍だが既定では自動ベクトル化と並ぶ。Vector API の価値は保証性。
6. **結果③強スケーリング**：≤4スレッドで ~0.99、以降は周波数スケーリングで低下。
7. **結果④GPU**：Adreno を Vulkan compute で叩けることを実証（CPU 一致検証つき）。Rémy 素朴移植の限界と
   unranking という打開策（適用範囲 n≤35）。
8. **今後**：三角形→デルタ規則の確定 → 本番評価器 → GPU 実測。

**測定はノート実測につき、効率などの絶対値は誤差込みで解釈し単一値で断定しない**、という誠実さを一貫させると良い。

## 9. 実行方法（再現用）

```powershell
# 単体テスト（9件、正しさの規律）
./mvnw test

# JMH 本測定（強スケーリング＋ゼロアロケーション）
./mvnw clean test-compile
./mvnw dependency:build-classpath "-Dmdep.outputFile=target/cp.txt" "-Dmdep.includeScope=test"
$CP = "target/test-classes;target/classes;$(Get-Content target/cp.txt)"
java -cp $CP org.openjdk.jmh.Main "StrongScalingBenchmark|ViolationSamplerBenchmark" `
  -f 5 -wi 5 -i 10 -w 3 -r 3 -jvmArgsAppend "-Xms8g -Xmx8g" -prof gc

# JMH SIMD（Vector API のフラグを同じ文字列にまとめる。-jvmArgsAppend は @Fork を上書きする）
java -cp $CP org.openjdk.jmh.Main "ReferenceSetAlgebraBenchmark" `
  -f 5 -wi 5 -i 10 -w 3 -r 3 -jvmArgsAppend "--add-modules=jdk.incubator.vector -Xms8g -Xmx8g" -prof gc
# 対照実験は上に " -XX:-UseSuperWord" を足す

# GPU 最小カーネル
java --enable-native-access=ALL-UNNAMED -cp "target/classes;$(Get-Content target/cp.txt)" `
  dev.shoheiyamagiwa.enumerator.gpu.Mix64GpuProbe
```
