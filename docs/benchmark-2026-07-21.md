# ベンチマーク測定記録 2026-07-21（安定化版）

[benchmark-2026-07-20.md](benchmark-2026-07-20.md) の不安定な初回測定を、方法論を統制して取り直した
確定版。**この版の数値を正とする。** 07-20 版は誤差が最大 ±43% あり、ゼロアロケーションの結論が
測定アーティファクトで逆転していた（下記「07-20 版からの訂正」参照）。

## 測定環境

| 項目 | 値 |
|---|---|
| CPU | Snapdragon X Elite - X1E78100 (Qualcomm Oryon)、12コア / 12論理プロセッサ |
| メモリ | 31.6 GB |
| OS | Windows 11 Home ARM 64-bit |
| JDK | OpenJDK 25.0.1+8-LTS (`jdk-25.0.1.8-arm`)、OpenJDK 64-Bit Server VM |
| JMH | 1.37 |
| ベクタ種 | `LongVector.SPECIES_PREFERRED` = 128bit / 2レーン（ARM64 NEON、SVE なし） |

ノート PC 上の測定。電源プランを「高パフォーマンス」に変更して実施したが、サーマル・周波数スケーリングは
統制していない（強スケーリングの効率解釈に影響。後述）。

## 07-20 版からの安定化

| 変更 | 目的 |
|---|---|
| `-f 2` → `-f 5` | フォーク間ばらつき（JVM 起動ごとの JIT 当たり外れ）を平均に反映 |
| ウォームアップ/測定 各1秒 → 各3秒 | 1測定あたりの母数を増やし瞬間ノイズを希釈 |
| ヒープ非固定 → `-Xms8g -Xmx8g` | ヒープ拡張イベントと GC 頻度変動を抑制 |
| `-prof gc` 追加 | 割り当て速度・GC を定量化（ゼロアロケーション主張の直接検証） |

`-prof gc` の実測により、8g で十分（`gc.count` は遅い基準でも1測定あたり1〜2回、高速経路は0回）と確認。

## 実行条件

共通: `@Fork(5)`、warmup 5×3s、measurement 10×3s。生データ:

- `jmh-stable-1.json` — 強スケーリング＋ゼロアロケーション（Vector API 不要）
- `jmh-stable-2.json` — SIMD 通常設定（自動ベクトル化オン）
- `jmh-nosuperword-202607202016.json` — SIMD 対照実験（`-XX:-UseSuperWord`、自動ベクトル化オフ）。
  07-20 の測定だが誤差 0.2% と安定していたため流用。

```powershell
# 強スケーリング＋ゼロアロケーション
java -cp $CP org.openjdk.jmh.Main "StrongScalingBenchmark|ViolationSamplerBenchmark" `
  -f 5 -wi 5 -i 10 -w 3 -r 3 -jvmArgsAppend "-Xms8g -Xmx8g" -prof gc `
  -rf json -rff docs/jmh-stable-1.json

# SIMD 通常設定
java -cp $CP org.openjdk.jmh.Main "ReferenceSetAlgebraBenchmark" `
  -f 5 -wi 5 -i 10 -w 3 -r 3 -jvmArgsAppend "--add-modules=jdk.incubator.vector -Xms8g -Xmx8g" -prof gc `
  -rf json -rff docs/jmh-stable-2.json
```

> **注意**: JMH の CLI `-jvmArgsAppend` は `@Fork(jvmArgsAppend=...)` を**上書き**する（追加ではない）。
> Vector API を使う SIMD ベンチでは `--add-modules=jdk.incubator.vector` を同じ文字列にまとめないと
> `NoClassDefFoundError: jdk/incubator/vector/Vector` で落ちる。

## 結果1: ゼロアロケーション化（最も強い結果）

`ViolationSamplerBenchmark`、1標本あたりのスループット。

| n | 遅い基準 (ops/s) | 高速経路 (ops/s) | 高速化 | 遅い基準 alloc | 高速経路 alloc |
|---:|---:|---:|---:|---:|---:|
| 12 | 746,623 ± 17,383 | 1,538,632 ± 1,550 | **2.06x** | 3,709 B/op | ~0.002 B/op |
| 30 | 231,448 ± 682 | 589,754 ± 563 | **2.55x** | 8,992 B/op | ~0.005 B/op |
| 60 | 97,780 ± 105 | 288,946 ± 150 | **2.96x** | 17,816 B/op | ~0.012 B/op |

- **高速経路が全 n で有意に速く（信頼区間は重ならない）、高速化は n とともに 2.06→2.96 倍に拡大する。**
- `-prof gc` が「ゼロアロケーション」を直接実証: 高速経路の `gc.count = 0`（GC 一切なし）、割り当ては
  約 0.002〜0.012 B/op ＝ 実質ゼロ（残りは JMH 基盤のノイズ）。遅い基準は n に比例して 3.7〜17.8 KB/op を
  捨てており（`ArrayList` + `int[3]`×n + 内部配列群）、GC を 50〜80 回起こしている。
- **相互検証**: 単一スレッドの並列サンプリング（n=60）695ms/op = 287,770 標本/秒は、独立ベンチの高速経路
  288,946 ops/s と 0.4% 差で一致。二つのベンチが内部整合している。

### 07-20 版からの訂正

07-20 版は n=12 で「遅い基準 1,001,074 対 高速経路 859,491 ＝ 遅い基準の方が速い」とし、「主張が崩れた」
と結論していた。これは**不安定測定によるアーティファクトで、完全な誤り**だった。安定化後は高速経路が
2.06 倍速い。当時の相互検証（並列 379k 対 独立ベンチ 243k）が食い違っていたことが、07-20 の数値が
信用できなかった証拠。**「段階0（測定安定化）を GPU より先に行う」判断がこれで正当化された。**

## 結果2: SIMD（因果が確定）

`ReferenceSetAlgebraBenchmark`、`(use1 | use2) & ~def1` のスカラー対 Vector API。

### 通常設定（自動ベクトル化オン、実運用の姿）

| 参照数 | scalar (ns/op) | vector (ns/op) | S/V |
|---:|---:|---:|---:|
| 64 | 3.545 ± 0.009 | 3.649 ± 0.002 | 0.97 |
| 256 | 6.691 ± 0.003 | 6.098 ± 0.002 | 1.10 |
| 1024 | 11.187 ± 0.031 | 10.360 ± 0.011 | 1.08 |
| 4096 | 27.076 ± 0.010 | 26.719 ± 0.820 | 1.01 |
| 16384 | 91.784 ± 0.165 | 89.394 ± 0.032 | 1.03 |
| 65536 | 373.082 ± 0.141 | 369.290 ± 0.144 | 1.01 |

**Vector API の上積みはほぼゼロ（1.0〜1.1倍）。** 07-20 で ±28〜43% の誤差の中に埋もれていた「差が出ない」
という事実が、安定化（誤差 0.2% 以下）後も揺るがず確定した。

### 対照実験（`-XX:-UseSuperWord`、自動ベクトル化オフ、NEON の素の実力）

| 参照数 | scalar (ns/op) | vector (ns/op) | S/V |
|---:|---:|---:|---:|
| 64 | 2.679 ± 0.009 | 2.758 ± 0.010 | 0.97 |
| 256 | 4.155 ± 0.008 | 4.609 ± 0.015 | 0.90 |
| 1024 | 10.231 ± 0.068 | 7.807 ± 0.024 | 1.31 |
| 4096 | 32.681 ± 0.449 | 19.529 ± 0.034 | 1.67 |
| 16384 | 121.941 ± 0.971 | 67.480 ± 0.121 | 1.81 |
| 65536 | 483.191 ± 0.938 | 265.019 ± 9.800 | **1.82** |

**参照集合が大きいほど NEON の理論上限2倍に単調に近づき、1.82倍で頭打ち。**

### scalar を ON/OFF で比較（C2 自動ベクトル化の効果を直接可視化）

| 参照数 | ON (ns/op) | OFF (ns/op) | OFF/ON |
|---:|---:|---:|---:|
| 4096 | 27.076 | 32.681 | 1.21 |
| 16384 | 91.784 | 121.941 | 1.33 |
| 65536 | 373.082 | 483.191 | 1.30 |

自動ベクトル化を切ると scalar が 1.2〜1.3倍遅くなる ＝ **C2 がスカラーループを実際に SIMD 化している
動かぬ証拠**。

### 結論

> Vector API は NEON で **1.82倍**出せる（対照実験）。しかし C2 の自動ベクトル化が同じ仕事を
> スカラー版に効かせている（**1.3倍**分）ため、通常設定では両者がほぼ並ぶ。
> **Vector API の価値は速度ではなく、自動ベクトル化の適用可否に依存せず SIMD 化を保証できること。**

補足: 全条件で `gc.count = 0`・割り当て 10⁻⁶〜10⁻⁴ B/op（事前確保バッファに書くだけ）で、測定が
アロケーションに汚染されていないことを確認。

## 結果3: 強スケーリング（数値は堅いが、効率の解釈に注意）

`StrongScalingBenchmark`、deltaCount=60、S=200,000 標本を固定してスレッド数を変化。

| threads | ms/op | speedup | efficiency |
|---:|---:|---:|---:|
| 1 | 695.03 ± 2.64 | 1.00 | — |
| 2 | 348.21 ± 0.77 | 2.00 | 0.998 |
| 4 | 175.00 ± 0.29 | 3.97 | 0.993 |
| 8 | 100.38 ± 1.76 | 6.92 | 0.866 |
| 12 | 82.22 ± 1.43 | 8.45 | 0.704 |

- **2〜4スレッドの効率はほぼ完璧（0.998, 0.993）。** 設計間が独立（embarrassingly parallel）であることを裏づける。
- **8スレッドで 0.866、12スレッドで 0.704 と、4スレッドを境に崖のように落ちる。** 並列化オーバーヘッドが
  原因なら緩やかに劣化するはずで、崖にはならない。**最有力の説明は CPU 周波数スケーリング**：
  1〜数コアなら高クロックにブーストするが、全12コア稼働時は基底クロックに落ちる。効率低下の大部分は
  同期コストではなくハードウェアの周波数低下と考えられる（崖の位置 4→8 が全コア稼働の閾値と整合）。
- **効率は単一値で語れない。** 07-20 版では 8スレッドが 0.991 だったが本版は 0.866。これは基準 t=1 が
  どれだけブーストしたかに効率が左右されることを示す。CLAUDE.md の「効率~0.97」は撤回すべき。
- 断定には周波数固定または実クロック測定が必要。発表では「≤4スレッドで効率0.99を確認、それ以上の
  低下は周波数スケーリングが支配的」と条件付きで述べるのが誠実。

## 段階0（測定安定化）の総括

- **4本すべての結果が発表可能な精度になった**（強スケーリング／ゼロアロケーション／SIMD 通常・対照）。
- 安定化はゼロアロケーションの結論を**逆転**させた。不安定な数値の上に GPU を積む前に足元を固める、
  という順序判断が正しかったことを示す。
- 残る宿題（この測定の範囲外）:
  - 強スケーリングの効率低下を周波数スケーリングで確定させる（クロック測定 or 固定）。
  - SIMD・評価器は現状パイプライン未接続の単体測定。三角形→デルタ規則の確定 → 接続アダプタが入れば、
    評価が集合演算になり SIMD がクリティカルパスに乗る。

## CLAUDE.md への反映が必要な箇所

- 「実機12コアで効率~0.97」→ 撤回。効率は基準のブースト状態に依存し単一値で語れない。
  実測は 2〜4スレッドで ~0.99、8で 0.87、12で 0.70。
- 「SIMD ... 参照集合が大きいほど理論上限（約2倍）に近づく」→ **自動ベクトル化を無効化した場合に限り**
  1.82倍。既定構成では C2 の自動ベクトル化により差はほぼ出ない。
