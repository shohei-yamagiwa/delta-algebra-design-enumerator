> **⚠️ この記録は [benchmark-2026-07-21.md](benchmark-2026-07-21.md) に置き換えられた（SUPERSEDED）。**

> フォーク間ばらつきと JIT 変動により誤差が最大 ±43% に達していた。特に「結果3: ゼロアロケーション化

| 項目 | 値 |

## 実行条件

共通: `@Fork(2)`、warmup 5 iterations × 1s、measurement 10 iterations × 1s
（`StrongScalingBenchmark` のみ warmup 3 / measurement 5）。

```powershell
# 通常設定
java -cp $CP org.openjdk.jmh.Main -rf json -rff target/jmh-result.json

# 対照実験（C2 の自動ベクトル化を無効化）
java -cp $CP org.openjdk.jmh.Main ReferenceSetAlgebraBenchmark `
  -jvmArgsAppend "--add-modules=jdk.incubator.vector -XX:-UseSuperWord" `
  -rf json -rff target/jmh-nosuperword.json
```

生データ: `jmh-result-202607202007.json`（通常設定）、`jmh-nosuperword-202607202016.json`（対照実験）。

> **注意**: JMH の CLI `-jvmArgsAppend` は `@Fork(jvmArgsAppend = ...)` を**上書き**する（追加ではない）。
> `-XX:-UseSuperWord` だけを渡すと `--add-modules=jdk.incubator.vector` が失われ、
> `NoClassDefFoundError: jdk/incubator/vector/Vector` で落ちる。必ず両方を1つの文字列で渡すこと。

## 結果1: 強スケーリング（最も信頼できる）

`StrongScalingBenchmark`、deltaCount=60、S=200,000 標本を固定してスレッド数を変化。

| threads | ms/op | speedup | efficiency |
|---:|---:|---:|---:|
| 1 | 527.69 ± 4.08 | — | — |
| 2 | 259.94 ± 2.54 | 2.03 | 1.015 |
| 4 | 137.27 ± 11.41 | 3.84 | 0.961 |
| 8 | 66.56 ± 1.95 | 7.93 | 0.991 |
| 12 | 49.75 ± 4.53 | 10.61 | 0.884 |

**CLAUDE.md の「12コアで効率~0.97」は支持されない。実測は 0.884。**
8スレッドで 0.991 と極めて良く、12スレッドで落ちるのは全コア占有時の OS / JMH オーバーヘッドとして自然。

留意点:

- 2スレッドの効率 1.015 はわずかに超線形。信頼区間はぎりぎり重ならない。スレッドあたりの
  作業集合が縮んでキャッシュ効率が上がった可能性があるが、断定はできない。
- 12スレッドは fork1 平均 47.3ms、fork2 平均 52.2ms と**フォーク間で約10%の系統差**がある。
  サーマルまたは周波数制御を疑う。フォーク数を増やして確認する必要がある。

## 結果2: SIMD（仮説は確認、ただし既定構成では差が出ない）

`ReferenceSetAlgebraBenchmark`、`(use1 | use2) & ~def1` のスカラー実装対 Vector API 実装。

### 対照実験（`-XX:-UseSuperWord`、自動ベクトル化オフ）

誤差はいずれも 1% 未満で、極めて安定している。

| 参照数 | scalar (ns/op) | vector (ns/op) | speedup |
|---:|---:|---:|---:|
| 64 | 2.679 ± 0.009 | 2.758 ± 0.010 | 0.97 |
| 256 | 4.155 ± 0.008 | 4.609 ± 0.015 | 0.90 |
| 1024 | 10.231 ± 0.068 | 7.807 ± 0.024 | 1.31 |
| 4096 | 32.681 ± 0.449 | 19.529 ± 0.034 | 1.67 |
| 16384 | 121.941 ± 0.971 | 67.480 ± 0.121 | 1.81 |
| 65536 | 483.191 ± 0.938 | 265.019 ± 9.800 | 1.82 |

**参照集合が大きくなるほど NEON の理論上限 2 倍に単調に近づき、1.82 倍で頭打ち。**
小さい集合（64/256）でベクタ版がわずかに負けるのは、ループ前処理のオーバーヘッドが償却されないため。

### 通常設定（自動ベクトル化オン）

| 参照数 | scalar (ns/op) | vector (ns/op) |
|---:|---:|---:|
| 64 | 3.035 ± 0.961 | 3.689 ± 1.361 |
| 256 | 10.744 ± 1.039 | 6.190 ± 0.712 |
| 1024 | 18.522 ± 5.225 | 7.839 ± 0.137 |
| 4096 | 21.122 ± 0.534 | 19.777 ± 0.245 |
| 16384 | 80.672 ± 26.559 | 172.375 ± 68.952 |
| 65536 | 376.209 ± 163.379 | 280.714 ± 15.881 |

スカラー版を両設定で比較すると、自動ベクトル化の効果が直接見える:

| 参照数 | scalar (オン) | scalar (オフ) | 自動ベクトル化による改善 |
|---:|---:|---:|---:|
| 4096 | 21.122 | 32.681 | 1.55x |
| 16384 | 80.672 | 121.941 | 1.51x |
| 65536 | 376.209 | 483.191 | 1.28x（誤差大） |

### 結論

**C2 はスカラーループを実際に自動ベクトル化しており、それが Vector API に上積みが出ない原因である。**
したがって主張は次の形に組み替えるべき:

> 「Vector API で 2 倍速くなる」ではなく、
> **「NEON で 1.82 倍が出せることを示した上で、C2 の自動ベクトル化が同じ仕事をしているため既定構成では
> 差が出ない。Vector API の価値は速度ではなく、自動ベクトル化の適用可否に依存せず SIMD 化を保証できること」**

### 未解決: 通常設定の測定が不安定

**通常設定の数値はそのままでは発表に使えない。**

| | 通常設定の相対誤差 | UseSuperWord オフ |
|---|---:|---:|
| scalar@1024 | ±28% | ±0.7% |
| scalar@65536 | ±43% | ±0.2% |
| vector@16384 | ±40% | ±0.2% |

生データは二峰性を示す。scalar@1024 は fork1 が一貫して 22〜23ns なのに対し、fork2 は 8.4ns で始まり
20ns まで上昇する。vector@16384 は 67ns から 368ns まで約6倍振れる。
**自動ベクトル化を切ると全て安定する**ため、C2 の SuperWord パスがフォークごと・コンパイルごとに
適用されたりされなかったりしている疑いが濃い。この現象自体が報告に値する可能性がある。

## 結果3: ゼロアロケーション化（主張が崩れた）

`ViolationSamplerBenchmark`、1標本あたりのスループット。

| deltaCount | 遅い基準 (ops/s) | ゼロアロケーション (ops/s) | 判定 |
|---:|---:|---:|---|
| 12 | 1,001,074 ± 4,029 | 859,491 ± 62,214 | **遅い基準の方が有意に速い** |
| 30 | 303,594 ± 2,287 | 403,521 ± 160,114 | 誤差過大、判定不能 |
| 60 | 55,268 ± 22,328 | 243,191 ± 76,601 | 誤差過大、方向のみ |

**n=12 では信頼区間が重ならず、遅い基準の方が有意に速い。** 小さい n では TLAB 割り当てが十分安く、
`eval` 冒頭で毎回行う配列クリアの方が高くつく、という筋の通った結果。

n=60 は方向としてはゼロアロケーション側が有利だが、誤差40%では数字を出せない。
遅い基準の n=60 が fork 内で 28k〜128k ops/s と 4.5 倍振れているのは、ほぼ GC が原因と考えられる
（1標本あたり `ArrayList` + `int[3]`×60 + 配列7本を捨てている）。

スモークテストで見えた「約3倍」は**再現しなかった**。

## この測定から導かれる次の一手

**GPU (Adreno + Vulkan) より先に、測定の安定化を優先すべき。**
現時点で発表にそのまま出せるのは強スケーリングのみで、SIMD は対照実験側しか使えず、
ゼロアロケーションは主張が崩れている。足元が固まらないうちに GPU を積んでも、
同じ理由で信用できない数字になる。

安定化の具体策（半日程度）:

1. `-f 5` 以上にフォーク数を増やし、フォーク間系統差を平均に埋もれさせずに評価する
2. `-Xms4g -Xmx4g` でヒープを固定し、GC 頻度の変動を抑える
3. `-prof gc` で割り当て速度と GC 時間を定量化する（結果3の裏取り）
4. `-prof perfnorm` が ARM64 Windows で動くか確認し、動けば命令数・キャッシュミスで
   結果2の二峰性を切り分ける
